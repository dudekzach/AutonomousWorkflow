Phase Three  ·  The Grind

What Real Building Actually Looks Like

I want to be honest about this stretch, because I think sanitized building stories do a disservice to anyone in their own hard chapter right now.

The deployed system kept returning one maddening response: HTTP 502. The workflow that ran flawlessly in VS Code was dying silently in production. I didn't know where. I didn't know why.

⚡ The wall

Claude alone was taking 40–50 seconds to respond. Add OpenAI's 10–15 seconds, the judge's 3–4 seconds, and overhead: a 60–70 second synchronous HTTP request held open while Render's load balancer waited, then timed out and killed it. My system wasn't broken. My architecture was fundamentally incompatible with production web infrastructure.

I debugged methodically. First I implemented a proper /health endpoint so Render could confirm the service was alive at all. Then I added structured logging layer by layer — entry points, input size, timing, errors, completion. Once I could see what was happening, I could fix it.

The specific crash revealed itself through the logs: my save_outputs() function was trying to join Python dictionary objects as if they were strings — a silent TypeError. A path alignment bug had files saving to a relative path while FastAPI's static server looked in an absolute one. These aren't glamorous problems. They're the invisible, character-building problems every serious builder has to sit through.

Phase Four  ·  The Architectural Leap

Throwing Out My Architecture and Starting Smarter

This is the chapter I'm most proud of. Not because the solution was elegant — though it was — but because of what it required: the willingness to look at working code and say, this is fundamentally wrong, and I need to rebuild it.

The synchronous architecture wasn't just slow. It was a dead end. I redesigned the entire execution model from the ground up.

Before
After
In-memory state
Redis-backed persistence
Synchronous execution
Async background worker
Single blocking request
job_id returned immediately
No state across restarts
Jobs survive server restarts
Models run sequentially
OpenAI + Claude run in parallel
~60–70s total runtime
~12s average runtime
One process, no scale
Horizontally scalable

Redis became the spine of everything. Jobs stored externally meant my system could survive a server restart without losing in-flight work. A job_id returned immediately meant the browser never waited on me.

Then came the insight that cut runtime by over 80%: Why are OpenAI and Claude running one after the other? They're independent. Only the judge needs both to finish. So I made them run simultaneously — each racing while the judge waited. Runtime dropped from 70 seconds to 12. Not from faster hardware or premium API access. From thinking clearly about which operations actually depend on each other.

That's not a beginner move. That's system design. And I figured it out by staring at a problem long enough to finally understand it.

Something I Noticed in the Data

My Judge Doesn't Just Score — It Reasons

One of the most interesting things I found when reviewing my run logs: the exact same prompt — "Write a short paragraph about customer service" — was submitted twice on April 24, seven minutes apart. The first time, OpenAI won. The second time, Claude won.

This isn't a bug. At 15:06, the judge penalized Claude for over-delivering — the request asked for "a short paragraph" and Claude produced a structured multi-section response. OpenAI stayed concise and was rewarded for fitting the brief. Seven minutes later, the judge weighted depth and engagement more heavily, and Claude's richer answer earned the win.

💡 What this means

My judge isn't applying a static rubric. It's applying contextual reasoning — evaluating fit as much as quality. Non-deterministic output from a reasoning engine is a feature, not a bug. It means the judge is actually thinking about the prompt, not just the response.
